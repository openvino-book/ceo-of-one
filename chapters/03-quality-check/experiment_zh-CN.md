# 第三章实验：给贪吃蛇游戏加上质量检查清单

## 目标

评估在构建阶段之前添加验收标准/质量检查清单，相比第二章使用的临时方法，是否能减少构建后的问题。

## 实验设置

| 项目 | 详情 |
|---|---|
| **基础任务** | 用与第二章相同的需求构建贪吃蛇游戏 |
| **COO 角色** | OpenClaw（相同模型，相同会话） |
| **工程师** | Claude Code（相同配置） |
| **变量** | 构建前定义验收标准 vs. 无标准 |
| **技术栈** | HTML + CSS + 原生 JavaScript |
| **方法** | 相同的提示词，相同的工程师——唯一的区别是检查清单 |

## 过程

### 阶段 A：无检查清单构建（复现第二章）

1. CEO 发出："帮我做个贪吃蛇游戏"——无验收标准
2. COO 分解为基本需求（网格、蛇移动、食物、分数）
3. 工程师构建游戏
4. COO 目视确认并接受

### 阶段 B：有检查清单构建

1. CEO 发出："帮我做个贪吃蛇游戏"——完全相同的请求
2. COO 分解为相同的基本需求
3. **新增：** COO 在发送给工程师之前添加了 10 项验收标准：
   - 构建通过（控制台零错误）
   - 蛇在边界穿越（指定行为）
   - 分数正确递增
   - 游戏结束正常触发并能干净重置
   - 代码中无 TODO/FIXME/console.log
   - 布局在 375px（手机）和 1280px（桌面）下正常
   - 食物不会生成在蛇身上
   - 游戏速度可玩（不快不慢）
   - 键盘和触摸控制都能用
   - 页面标题和基本样式完整
4. COO 将需求+检查清单发送给工程师
5. 工程师构建游戏
6. COO 逐项对照检查清单验证

## 结果

| 指标 | 无检查清单（第二章） | 有检查清单（第三章） |
|---|---|---|
| 构建错误 | 0 | 0 |
| 控制台警告 | 2（未使用变量） | 0 |
| TODO/FIXME 注释 | 3 | 0 |
| 移动端布局问题 | 有 | 无 |
| 边界情况（食物在蛇身上） | 未处理 | 已处理 |
| COO 需要返工次数 | 1（修复移动端） | 0 |
| 总迭代次数 | 2 | 1 |
| COO 耗时 | ~8 分钟 | ~6 分钟（返工更少） |

## 关键发现

**1. 检查清单迫使 COO 提前思考质量。** 在构建之前写下"布局在 375px 下正常"意味着工程师从一开始就收到了这个需求。在第二章中，这是事后才发现的。

**2. 零返工 = 更短的总时间。** 第二章用了 2 次迭代（构建→发现移动端坏了→修复）。第三章只用了 1 次。花 2 分钟写检查清单，节省了 5 分钟返工。

**3. "未处理"的边界情况是最常见的 bug。** 食物刷在蛇身上是经典问题：工程师想不到是因为没人告诉他们。检查清单项"食物不会生成在蛇身上"让它变得明确。工程师第一次就正确实现了。

**4. 控制台警告是沉默的质量债务。** 第二章中的 2 个未使用变量警告不是 bug——暂时还不是。它们表明工程师留下了探索代码。"无控制台警告"这条标准立刻抓住了它们。

**5. 检查清单防止 COO 偷懒。** 没有它，COO 的倾向是扫一眼结果，说"看着不错"，然后继续。检查清单迫使进行有纪律的、系统性的验证。它把主观的"我觉得没问题"变成客观的"10/10 标准达标"。

## 这证明了什么

验收标准不是额外负担。它们是省时工具。花 2 分钟写检查清单，减少了约 25% 的总构建时间（从 8 分钟降到 6 分钟），并消除了所有构建后返工。

对于 CEO of One 方法来说，这是质量控制层：**在构建之前，定义"完成"意味着什么；构建之后，对照那个定义来验证。** 之后每一章都遵循这个模式。

## 沉淀知识

- ✅ 始终在 SPECIFY 步骤之前定义验收标准
- ✅ 至少包含：构建干净、无控制台警告/错误、无 TODO/FIXME、响应式检查
- ✅ 逐项验证——不要"整体看着不错"
- ✅ 检查清单能抓住工程师自己想不到的边界情况
- ✅ 没有检查清单 = 没有标准 = 主观验收 = 不可靠的质量
