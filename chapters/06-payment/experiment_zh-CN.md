# 第六章实验：支付系统

## 目标

在现有平台上添加模拟支付系统——结账（创建订单）、验证（完成支付）、报名追踪。将支付与认证（用户身份）和课程（购买内容）连接起来。这是**第三个功能模块**，也是第一个整合了*两个*现有模块数据的模块。

## 初始状态

- **起始状态：** 模块化平台，含课程模块（34个测试）+ 认证模块（31个测试）= 65个通过测试
- **任务：** 添加支付模块——结账、支付验证、报名追踪
- **约束：** 不修改 `src/features/courses/` 或 `src/features/auth/`
- **可用依赖：** Express、TypeScript、Jest、现有共享类型、courseStore、authStore

## CEO 输入

> 继续阶段八

一句话。COO 将其分解为完整的支付模块规格：3个API端点、11个验收标准。

## 构建内容

### 支付模块 (`src/features/payment/`)
| 文件 | 用途 |
|------|------|
| `types.ts` | 支付相关类型定义 |
| `store.ts` | 内存订单存储 |
| `service.ts` | 业务逻辑——结账、验证、报名 |
| `routes.ts` | 带输入验证的 Express 路由 |
| `payment.test.ts` | 29个测试，覆盖所有端点和边界情况 |

### API 端点
| 端点 | 方法 | 描述 |
|------|------|------|
| `/api/payments/checkout` | POST | 为课程创建订单 |
| `/api/payments/verify` | POST | 完成支付并报名 |
| `/api/payments/enrollment` | GET | 列出用户已报名的课程 |

### 共享类型修改
在 `src/lib/types/index.ts` 的 `AuthUser` 中添加了 `enrolledCourses` 字段。

### 依赖模式
支付模块**读取** `courseStore`（商品数据）和 `authStore`（用户身份），但**从不修改**它们。这是单向依赖规则：payment → lib → courses/auth。

## Bug 记录

**Bug 1 — `routes.ts` 中的可区分联合类型访问：**
- **问题：** 在未检查结果类型的成功/失败判别符的情况下访问 `result.enrolledCourses`
- **修复：** 在访问联合类型成功分支的字段之前添加了成功检查
- **结果：** 自动修复，重新构建，测试通过

**Bug 2 — `service.ts` 中的 `.map().filter()` 类型收窄：**
- **问题：** TypeScript 无法在链式 `.map().filter()` 调用中收窄类型
- **修复：** 拆分为两步——先 map，然后用收窄后的类型 filter
- **结果：** 自动修复，重新构建，测试通过

**Bug 3 — `types.ts` 中 `EnrolledCourse.category` 类型不匹配：**
- **问题：** `category` 类型为 `string`，但应为课程模块中的 `Category` 枚举类型
- **修复：** 导入 `Category` 类型并在 `EnrolledCourse` 定义中使用
- **结果：** 自动修复，重新构建，测试通过

## 关键洞察

### 1. 跨模块整合无耦合
支付读取 courseStore 和 authStore 但不修改它们。单向依赖规则（payment → lib → courses/auth）成立。这是第一个真正桥接两个现有功能的模块，架构干净地处理了它。

### 2. TypeScript 严格模式捕获真实 bug
3个 bug 全是类型不匹配，在运行时会报错。构建步骤不可或缺——这些正是 JavaScript 项目中容易漏掉的 bug。

### 3. 更多集成点 = 更多类型摩擦
3个 bug 对比第五章的1个。支付更复杂，因为它触及两个模块（认证用于用户身份，课程用于商品数据）。更多边界 = 更多类型不匹配的表面积。

### 4. 94个测试，2.3秒
测试套件依然很快。没有性能问题。每个模块添加自己的测试而不拖慢整体。

### 5. 模式已验证
第三个功能模块，零回归。模块化架构兑现了它的承诺。
