# 第六章：开始赚钱

**添加支付功能——把网站变成生意的那个功能。**

> 📋 [阅读完整实验记录](experiment_zh-CN.md)

---

## 🎯 你会学到

如何添加一个**连接两个现有模块**的新功能——认证（谁在买）和课程（买什么）——形成一个完整的支付系统。这是第一个新模块不只是自包含的章节；它需要*伸入*现有模块才能完成工作。

你还会看到复杂度上升时会发生什么：三个 bug，全部被编译器捕获，全部在人类毫不知情的情况下修复。

## 🗣️ CEO 说了什么

> 继续阶段八

一如既往。一句话。没有规格说明，没有技术选择，没有验收标准。COO 把这句话变成了一个完整的支付模块：三个 API 端点、十一个验收标准——以及一个硬性约束：不碰现有代码。

## 🔍 构建了什么

一个让已登录用户购买课程的支付系统。

工作流程是这样的：用户在课程上点击"结账"，创建一个待支付订单。然后"付款"（模拟的——没有真信用卡，没有真支付网关），订单变为已完成，课程加入报名列表。之后可以通过报名端点查看已购买的课程。

三个端点。一笔商业交易中的三个步骤。

这一章与前两章不同的是，支付模块需要知道用户**是谁**（来自认证模块）以及课程**多少钱**（来自课程模块）。它从两个模块读取数据，但从不修改任何一个。认证模块不知道支付的存在，课程模块也不知道。支付模块知道两者，但只是单向的。

这就是单向依赖规则：新功能可以查看现有功能，但现有功能永远不需要为了容纳新功能而改变。

## 🏗️ 三个 Bug，零恐慌

有趣的是：这一章有三个 bug，而上一章只有一个。代码变差了吗？

没有。代码做得更多了。

支付是第一个触及**两个**现有模块的模块，而不是一个。更多的集成点意味着更多跨模块边界的类型需要对齐的地方。每个边界都是潜在的不匹配，而 TypeScript 严格模式全部捕获了。

Bug 一：在未检查联合类型是哪个分支的情况下访问结果上的数据。Bug 二：TypeScript 无法在链式 `.map().filter()` 调用中追踪类型。Bug 三：一个字段被类型化为通用的 `string`，而它应该是课程模块中特定的 `Category` 类型。

三个 bug。三次自动修复。构建过程捕获了它们，代理修复了它们，CEO 从未察觉。这正是类型系统该做的事——在门口抓住错误，在它们变成运行时错误之前。

## 💡 模式已验证

看看这些数字：

- **第四章**（重构）：34个测试，0回归
- **第五章**（认证）：65个测试，0回归，1个 bug
- **第六章**（支付）：94个测试，0回归，3个 bug

每一章都添加一个完整的功能模块。每一章的现有测试都无需修改即可通过。测试套件在2.3秒内运行。零性能问题。

第四章的模块化架构现在已经深入三个功能，仍然完全按设计工作。功能模块加入，测试通过，什么都没坏。

## 🎓 章节要点

添加支付功能是大多数副项目停滞的地方。你有了用户，有了内容，但把"谁在买"、"买什么"和"钱到底有没有到账"连接起来，需要同时触碰系统的多个部分。在纠缠的代码库中，这就是回归爆发的地方。

在这个架构里，它只是另一个模块。

支付模块读取认证和课程但不修改它们。单向依赖规则意味着新功能到来时现有功能永远不需要改变。TypeScript 在构建时捕获不可避免的类型不匹配，而不是在凌晨三点在生产环境中。

三个功能。94个测试。零回归。CEO 的一句话。

系统在运转。
