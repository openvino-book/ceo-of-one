# 第九章 — 仪表盘与种子数据：实验记录

[English](experiment.md) | [中文](README_zh-CN.md)

## 任务

添加数据仪表盘和种子数据，让平台看起来像一个真正在运营的业务。

## COO 决策

**使用种子数据，不使用数据库。**

读者不需要学习数据库配置。仪表盘展示了真实业务仪表盘的样子，数据来自内存存储中预加载的演示数据。这样既保持了学习曲线的平缓，又展示了真实平台的指标长什么样。

## COO 流程

CEO 说："继续阶段十一"。

COO 执行了四个步骤：

1. **回顾** — 应用第七章的经验：写精确的规格说明、做结构检查、以零 bug 为目标。
2. **架构** — 决定使用种子数据 + 独立的仪表盘模块。仪表盘通过依赖注入读取其他模块的数据——没有跨功能导入，不使用数据库。
3. **规格** — 写了详细的规格说明，包括种子数据内容、仪表盘类型定义、页面布局设计和 15+ 测试的要求。
4. **执行** — Claude Code 根据规格构建了所有功能。

## 结果

| 指标 | 数值 |

| 构建错误 | **0**（开发过程中修复了 1 个 ESLint bug） |
| 测试 | **111/111 通过**（94 个已有 + 17 个新增仪表盘测试） |

## Bug 分析

**ESLint bug：** COO 的规格说"不允许跨功能导入"，但没有预料到 Claude Code 会在支付模块的种子数据函数中使用 `require()` 来访问其他模块（如课程和用户模块）。自动生成的 `eslint-disable` 注释引用了项目中不存在的规则，导致构建失败。

**根本原因：** 规格缺口。COO 应该明确禁止在种子数据函数中使用 `require()`（"种子数据也要使用依赖注入"），或者完全禁止 `eslint-disable` 注释。

**修复：** 删除了错误的 eslint-disable 注释。很简单，但在大量代码变更中很容易被忽略。

## 核心洞察

种子数据改变了产品的气质。第九章之前：空数组，一切都在说"这只是个演示"。之后：真实的中文名字、真实的课程标题、真实到 ¥1,124 的订单总额。同一个代码库——有了数据，它感觉像一家公司。

这就是为什么第八章"空课程"的时刻是第九章最完美的动机。你不曾感到空虚，就不会懂得数据的价值。
