> 仪表盘把一堆功能变成一家公司。

## 🎯 你会学到

数据把产品变成业务。一个功能完整但界面空空的应用，感觉像个原型。同样的应用加上真实的数字——用户、订单、收入——感觉就像一家公司。区别不在于代码，而在于数据。

这一章探讨的就是这种转变：种子数据如何把一个编程练习变成一个看起来和感觉上都像真正运营平台的产物。

## 🔍 构建了什么

一个暗色主题的仪表盘页面。顶部四个统计卡片：总用户数、总课程数、总订单数、总收入。下方两个表格：最近用户和最近订单。

数字是真实的——不是占位符，不是" lorem ipsum"。五个有中文名字的用户。六门有实际标题的课程。八笔订单，收入总计 ¥1,124。

幕后：一个 API 端点（`/api/dashboard`）从用户、课程和支付模块中聚合数据。全部由带预加载种子数据的内存存储驱动，不需要数据库。

测试套件从 94 个增长到 111 个——17 个新测试覆盖仪表盘 API 响应、种子数据完整性和页面渲染。

---

## 🏗️ 种子数据策略

关键的架构决策是：**不用数据库**。

COO 选择种子数据而非数据库，原因很简单——读者学的是构建产品，不是配置数据库。加 PostgreSQL 或 MongoDB 会让复杂度翻倍，但不会带来成比例的学习价值。

取而代之的是，每个模块（用户、课程、支付）都带有一个 `seed()` 函数，在服务器启动时用真实的演示数据填充内存存储。仪表盘模块通过依赖注入读取这些存储。

这行得通，因为仪表盘是只读的。它不创建、不更新、不删除任何东西，只负责观察。对于一个"瞭望台"来说，预加载数据再合适不过了。

结果：启动服务器，访问仪表盘，看到一家公司。不用配置，不用迁移，不用写 `INSERT INTO`。

## 💡 一个 ESLint Bug

开发过程中，Claude Code 遇到了构建错误。支付模块的种子数据函数需要课程和用户数据来创建真实的订单。但它没有通过依赖注入接收这些数据，而是用了 `require()` 直接导入了其他模块。

这违反了项目的依赖注入模式。Claude Code 试图用 `eslint-disable` 注释来抑制 ESLint 警告——但引用的规则名在项目中根本不存在。构建失败。

修复很简单：删掉那条注释。但教训很有意义。COO 的规格说了"不允许跨功能导入"，但没有明确覆盖种子数据函数。规格几乎是对的，而"几乎"依然产生了一个 bug。

**规则：** 如果一个模式很重要，就要在所有地方明确指定——包括数据初始化。规格中的缺口会导致实现中的缺口。

## 🎓 本章收获

同一个代码库，在种子数据前后，讲的是两个不同的故事。

之前：一个有功能但没有证据的平台。注册能用（但没有人注册）。支付能用（但没有人支付）。课程存在（零门）。技术上正确，情感上空洞。

之后：五个用户注册了。六门课程发布了。八笔订单下达了。¥1,124 的收入。平台不再让你想象——它直接展示正在发生的事。

数据就是证据。而仪表盘是一目了然的证据。

这一刻，项目不再是一个教程，而变成了一个产品。不是因为新功能。因为屏幕上的数字。
