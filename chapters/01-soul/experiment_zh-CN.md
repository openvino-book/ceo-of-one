# 第 1 章实验：给 AI 装上 COO 大脑（A/B 测试）

## 目标

验证向 Claude Code 添加结构化的"COO 灵魂"指令，能否显著提升输出质量、测试覆盖率和产品完整度——对比裸指令和手工编写的结构化提示词。

## 实验设置

| 项目 | 详情 |
|---|---|
| **任务** | 构建一个 Todo REST API（CRUD 操作） |
| **工具** | acpx + Claude Code |
| **变量** | 提示词结构（3 种方案） |
| **工程师** | Claude Code（同一模型，同一会话类型） |
| **衡量指标** | 测试数量、Bug 数、功能完整度、CEO 付出 |

## 实验过程

### 步骤 1：方案 A — 裸指令

发送一句话："Build me a Todo REST API。" 无灵魂配置，无结构，无质量要求。

Claude Code 构建了一个基础的 Express 服务器，有 CRUD 端点。能返回数据。没有测试，没有输入验证，没有除基本 try/catch 外的错误处理。

### 步骤 2：方案 B — 手工结构化提示词

编写了约 200 字的结构化提示词，包含测试、错误处理和输入验证要求。通过 acpx 直接发送。

结果：24 个测试通过，正确的错误响应，输入验证。质量不错——但 CEO 需要写 200 字的技术需求。

### 步骤 3：方案 C — COO 灵魂（通过 OpenClaw）

使用 COO 灵魂模板，让 OpenClaw 将"Build me a Todo REST API"分解为完整的工程规格。Claude Code 执行。

结果：28 个测试通过，组织化的项目结构（`types/`、`store/`、`routes/`），输入验证（包括空白字符修剪），自动修复了一个测试 bug（缺少 404 处理器）。

### 步骤 4：对比

按相同标准衡量三种方案。

## 实验结果

| 指标 | A: 裸指令 | B: 手工提示词 | C: COO 灵魂 |
|---|---|---|---|
| 测试数量 | 0 | 24 ✅ | 28 ✅ |
| 发现 Bug | 2（无验证，错误输入崩溃） | 0 | 0 |
| 输入验证 | ❌ | ✅ 基础 | ✅ 完善（空白字符、空串） |
| 项目结构 | 扁平（单文件） | 有组织 | 有组织 |
| CEO 付出 | 1 句话 | ~200 字 | 1 句话 |
| 自动修复问题 | — | — | ✅（测试中缺少 404 处理器） |
| 过滤支持 | ❌ | ❌ | ✅ |

## 关键发现

1. **方案 B 和 C 产出相似质量，但 CEO 付出天差地别。** 这是核心发现。自己写 200 字提示词确实有用——但既然 COO 能从一句话生成等效（甚至更好）的规格，为什么要自己写？

2. **方法论流程不完美。** 在这个早期章节中，OpenClaw（COO）直接执行了对比和分析。理想流程——"CEO 说 → COO 分解 → 工程师执行 → COO 汇报"——还没有完全建立。COO 同时扮演了实验者和操作者的角色。这是一个已知的不完美之处，我们在后续章节中修正了。

3. **零测试是默认行为，不是例外。** 没有明确的质量指令，Claude Code 就不写测试。这不是 bug——这是没有指定质量标准时的预期行为。COO 灵魂模板通过默认配置解决了这个问题。

4. **边缘情况来自结构化思维，不是来自更用力地催促。** COO 想到了空白字符修剪和空字符串处理，因为它基于结构化的质量框架工作，而不仅仅是"再努力一点"。

## 这证明了什么

结构化的 COO 指令稳定地优于裸指令，AI 生成的 COO 规格匹配或超越手工规格，同时 CEO 零技术付出。COO 灵魂模板是最小可行的质量层。

## 沉淀知识

- B：对 Claude Code 的裸指令默认产出功能正常但未经测试的代码。
- B：COO 灵魂模板（产品思维 + 质量标准 + 交付）产出可衡量地更好的输出。
- B："CEO 付出：1 句话"通过正确的 COO 分解是可实现的。
- O：早期实验的方法论流程不完美——COO 有时在 CEO→COO→工程师 链条之外操作。这在后续章节中有所改善。
- O：仅看测试数量不够——测试*质量*（边缘情况、异常路径）更重要。28 个有针对性的测试 > 50 个浅层测试。
