# 📋 你的 COO 有验收清单

> "确保它能用"和一份正式的验收清单之间的区别，就是原型和产品之间的区别。

---

## 🎯 你将学到

- 为什么"确保它能用"是软件里最贵的一句话
- 什么是验收标准（用大白话讲）
- 你的 COO 用的完整清单——以及为什么你永远不需要自己写
- 来自我们实验的真实数据：同一个任务，两种方式，截然不同的结果

---

## 🗣️ 你只需要说一句话

作为 CEO，你的工作只有一句话：

> "帮我建一个课程列表接口。"

完了。你的 COO 收到这句话，自动添加验收标准，工程团队就能清楚地知道什么叫"做完了"。

你不需要说"要处理错误"、"加上排序"、"校验输入"这些。COO 会做这些。这就是 COO 的职责。

---

## 🔍 为什么验收标准这么重要

我们做了一个实验。同样的任务（课程列表 API）、同样的工具、同样的团队——只是两种不同的方式。

### 结果对比

| | 无验收标准 | 有验收标准 |
|---|---|---|
| **用时** | ~5 分钟 | ~5.25 分钟 |
| **测试用例** | 16 个 | 23 个 |
| **发现的 Bug** | 2 个 | 0 个 |
| **筛选功能** | ❌ | ✅ |
| **排序功能** | ❌ | ✅ |
| **输入校验** | ❌ | ✅ |
| **结构化错误响应** | ❌ | ✅ |

再读一遍。**多用了 15 秒。** 结果好了十倍。

### 没有标准时出了什么问题

当你说"确保它能用"时，团队会按照*他们自己*的理解来构建"能用"的定义。而他们的定义通常跟你的不一样。他们可能跳过错误处理，因为"正常使用时没问题"。他们可能跳过排序，因为"数据能显示出来"。他们可能跳过校验，因为"前端会处理"。

每一个被跳过的项目，都是一个等待爆发的 Bug。每一个假设，都是一场等待浮出水面的分歧。

## 📋 COO 的验收清单

下面是 COO 为"课程列表 API"这类任务添加的 13 条验收标准。你不需要记住它们，只需要知道它们存在。

1. **零编译错误** — 代码必须干净地构建。警告不当作可选项。
2. **至少 20 个测试用例** — 不是建议，是底线。覆盖每条路径、每个边界、每个异常场景。
3. **输入校验** — 字符串不能为空，价格不能为负数，分类必须在允许的范围内。
4. **按最新排序** — 结果按创建时间倒序排列。
5. **筛选支持** — 用户可以通过查询参数按分类和发布状态筛选。
6. **规范的 404 响应** — 不只是状态码，还要有 JSON 响应体说明缺少了什么。
7. **具体的校验错误** — 不要只说"输入无效"。要说*哪个字段*、*为什么*无效。
8. **统一的错误格式** — 每个错误都遵循相同的结构：错误类型、消息、可选的详细信息。
9. **不留捷径** — 没有 TODO 注释，没有 FIXME 标记，没有残留的调试日志。
10. **规范的目录结构** — 文件放在该放的位置，不把所有东西堆在一个文件夹里。
11. **分页支持** — 大量数据不应该一次性全部返回。
12. **边界情况覆盖** — 空结果、重复条目、名称中的特殊字符。
13. **文档** — 简要说明接口的功能和使用方法。

这就是清单。COO 来写，团队来执行，你得到的是产品，不是原型。

## 💡 秘密在这里

"确保它能用"听起来很负责任。听起来像是在对团队提出要求。

其实不是。恰恰相反。

"确保它能用"是模糊的。模糊的需求产生模糊的结果。团队*确实*确保了它能用——按照他们的定义。而他们的定义是：只要一切正常，代码能跑，数据能返回就行。

验收标准是具体的。具体的需求产生具体的结果。团队清楚地知道"完成"是什么样子。不用猜、不用假设、不用意外。

关键来了：**你不需要自己写验收标准。** 这是 COO 的工作。COO 把你的一句话需求变成详细、无歧义的规格说明。那多出的 5% 时间（在我们的实验中是 15 秒）能给你换来：

- 44% 更多的测试覆盖
- 零 Bug 而不是两个
- 你以后还得提需求才能加上的功能（筛选、排序、校验）
- 能告诉用户到底出了什么错的错误处理

COO 不会拖慢你。COO 确保你不用做两遍。
