# 第 3 章实验：验收标准与质量检查清单

## 目标

验证结构化的验收标准（质量检查清单）是否比模糊的"确保能用"指令显著提升输出质量——以及 COO 能否自动生成这些标准。

## 实验设置

| 项目 | 详情 |
|---|---|
| **任务** | 构建一个课程列表 REST API（CRUD、筛选、排序） |
| **工具** | acpx + Claude Code + COO 灵魂模板 |
| **变量** | 是否附加验收标准（无 vs. 结构化清单） |
| **衡量指标** | 测试数量、Bug 数、功能完整度、耗时 |

## 实验过程

### 步骤 1：方案 A — "确保能用"

使用 COO 灵魂模板发送任务，但没有具体的验收标准，只有隐含的"确保能用"。

Claude Code 构建了一个功能正常的 API。测试通过。但：
- 只有 16 个测试用例（有标准的版本是 23 个）
- 没有输入验证（空字符串、负价格、无效分类直接通过）
- 没有筛选支持
- 没有排序支持
- 错误响应只有裸状态码，没有 JSON 体

**耗时：** 约 5 分钟

### 步骤 2：方案 B — COO 自动生成的验收标准

同样的任务，但 COO 在发送给工程师之前生成了 13 条验收标准：

1. 零编译错误
2. 至少 20 个测试用例
3. 输入验证（非空、正数价格、有效分类）
4. 按最新优先排序
5. 筛选支持（分类、发布状态）
6. 正确的 404 响应（带 JSON 体）
7. 具体的验证错误信息（哪个字段、为什么）
8. 一致的错误格式
9. 无 TODO/FIXME/console.log
10. 有组织的目录结构
11. 分页支持
12. 边缘情况覆盖（空结果、重复、特殊字符）
13. 文档说明

Claude Code 构建了同样的 API，但现在：
- 23 个覆盖边缘情况的测试用例
- 所有字段的输入验证
- 筛选和排序正常工作
- 结构化 JSON 错误响应
- 没有开发者遗留物

**耗时：** 约 5 分 15 秒

### 步骤 3：检查清单格式迭代

我们在确定最终模板之前尝试了三种清单格式：

| 格式 | 问题 | 结论 |
|---|---|---|
| 自由段落 | Claude Code 遗漏项目，没有明确的通过/失败 | ❌ 放弃 |
| 编号列表（1-13） | 更好，但项目之间感觉脱节 | ⚠️ 可用但不理想 |
| 分类+复选框 | 清晰的分组（构建、质量、用户体验、边缘情况） | ✅ **采用** |

分类格式成为了后续章节使用的 `review-checklist.md` 模板。

## 实验结果

| 指标 | 无标准 | 有标准 | 差异 |
|---|---|---|---|
| 耗时 | ~5:00 | ~5:15 | +15 秒 |
| 测试用例 | 16 | 23 | +44% |
| 发现 Bug | 2 | 0 | — |
| 输入验证 | ❌ | ✅ | — |
| 筛选支持 | ❌ | ✅ | — |
| 排序 | ❌ | ✅ | — |
| 结构化错误 | ❌ | ✅ | — |

## 关键发现

1. **15 秒。** 这就是验收标准的成本。额外的 15 秒换来零 Bug、44% 更多测试和三个额外功能。这个 ROI 简直离谱。任何反对"过度规格化"的论点在这个数字面前都站不住脚。

2. **"确保能用"是软件中最昂贵的句子。** 听起来负责任，但实际上是最懒惰的质量标准。团队确实"确保能用了"——按他们的定义（编译通过，正常输入能返回数据）。你的定义不同。验收标准弥合了这个差距。

3. **COO 应该生成标准，不是 CEO。** CEO 说"构建一个课程列表 API"。COO 自动添加 13 条标准。CEO 永远不需要思考分页或输入验证——COO 处理这些领域知识。

4. **清单格式不如清单存在本身重要。** 我们迭代了格式（段落 → 编号列表 → 分类），但最大的质量飞跃是从"没有清单"到"任何清单"。格式优化是 3.5 章的事。有清单本身就是第 3 章的发现。

5. **方法论仍在形成中。** 同样的早期章节说明：COO 运行了两种方案并做了对比。在成熟的流程中，这些应该是 COO 协调的独立工程师运行。我们自己在记录的同时还在学习这个工作流。

## 这证明了什么

结构化的验收标准是 ROI 最高的质量干预手段。成本 15 秒，消灭整类 Bug 和缺失功能。COO 可以自动生成，CEO 零注意力成本。我们确定的清单格式（分类+逐项通过/失败）成为了后续所有章节的永久模板。

## 沉淀知识

- B：验收标准增加约 15 秒构建时间，但产出 44% 更多测试覆盖和零 Bug。
- B："确保能用"不够——对不同的人意味着不同的事。
- B：COO 从 CEO 的一句话请求自动生成标准。
- B：分类检查清单格式（构建/质量/用户体验/边缘情况）对工程师最清晰。
- O：`review-checklist.md` 模板诞生于本次实验的格式迭代。
- O：清单格式迭代有影响，但清单*存在本身*的影响是它的 10 倍。
