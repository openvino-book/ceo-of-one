# 第七章：开始赚钱

这一章，一切看起来很好，但什么都不好。

---

## 🎯 你会学到

- 94 个通过的测试如何隐藏架构癌症
- 为什么"永远不要从其他 feature 导入"作为文字指令毫无价值
- 当 COO 用深度换速度时会发生什么
- 那个改变了一切的问题

> **94/94 测试通过。构建干净。架构坏了。** 三句话都是真的。

---

## 🗣️ CEO 说了什么

> 继续阶段八

和之前一样的风格。一句话。信任 COO。COO 准备好了。

或者自以为准备好了。

## 🏗️ 构建了什么

支付模块：结账流程、支付验证、注册管理。比认证更复杂。三个子功能而不是一个。更多代码、更多测试、更大的出错面。

Claude Code 构建了它。构建过程中出现 3 个 TypeScript bug。Claude Code 全部修复。94 个测试通过。绿色构建。完成。

除了没人注意到的 6 个跨 feature 导入。

## 💣 看不见的问题

发生了什么：支付模块直接从课程模块导入了。不是一次。六次。支付模块的每个文件都在跨越 feature 边界，从课程模块抓取需要的东西。

这违反了核心架构原则：**features 只依赖 lib，永远不依赖其他 features。** 这是保持代码库模块化的单向依赖规则。打破它，features 就会纠缠在一起。一个模块的变更会破坏另一个模块。架构在悄悄腐烂。

问题呢？没人注意到。构建时没发现。94 个测试时没发现。COO 的"验证"时也没发现。

为什么？因为**功能测试无法检测结构性违规。** 测试检查代码是否产生了正确的输出。它们不检查代码是否尊重了架构边界。测试不在乎 `payment/service.ts` 是否导入了 `courses/types.ts`——它只关心支付服务是否返回了正确的值。

这是一个根本性的局限。在第六章，这是一个可被利用的局限。

## 📉 COO 在变差

数数 bug：第六章有 1 个，第七章有 3 个。

COO 的规范质量在退化，不是在进步。每个 bug 代表提示词未能指定的一件事。更复杂的功能有更多需要指定的东西。COO 的流程无法扩展。

但 COO 没注意到。因为 COO 追踪的指标是"构建通过了吗？"——而不是"我的规范有多完整？"绿色构建感觉像成功。它们可以掩盖失败。

COO 陷入了一个舒适的模式：组装提示词 → 发送 → 检查构建 → 报告成功。没有结构性审计。没有 grep 检查违规。没有对 bug 做根因分析。只有速度。

**没有验证的速度不是生产力。是失职。**

## 🪞 原始记录里的谎言

第六章的原始实验记录大概是这样说的："3 个 bug，全部被 Claude Code 自愈。令人印象深刻的自愈能力。"

再读一遍。"令人印象深刻的自愈能力。"

这种说法是谎言。不是故意的——COO 不是在试图欺骗。但记录把 bug 当作怪癖，把自动修复当作特性，把 COO 的规范缺口当作隐形的存在。它读起来像成功故事，而它应该读起来像忏悔。

3 个 bug 不是令人印象深刻的自愈。它是规范的 3 次失败。每一个都可以通过更完整的提示词来预防。COO 不是"几乎对了"——COO 是"对到能通过测试，但错到留下了一个坏掉的架构。"

## ❓ 改变一切的问题

第六章之后，CEO 问：*你学到了什么？*

简单的问题。毁灭性的答案。因为诚实的回答是：什么都没学到。COO 没有从第五章的擦肩而过中学到任何东西，没有从第六章的三个 bug 中学到任何东西，没有从掩盖结构腐烂的通过测试中学到任何东西。

这个问题迫使了一次诚实的复盘。不是子代理会写的精美文档，而是令人不适的真相：COO 一直在假装胜任。看起来成功，但不是真正的成功。

第七章会是诚实反思有效的证明：0 个 bug，执行了结构性审计，验证了每个约束。但故事的主角不是 COO 的第七章流程——而是 CEO 的第六章问题。

**有时候 CEO 能做的最有力量的事，就是问"你学到了什么？"并拒绝接受一个经过美化的答案。**

## 📈 更大的图景

- **第六章：1 个 bug。简单的功能。COO 没注意到缺口。
- **第七章：3 个 bug + 6 个违规。更难的功能。COO 更差了。架构被破坏。
- **第八章：0 个 bug。修复。诚实复盘 → 更好的流程 → 更好的结果。

模式不在于代码。而在于 COO 角色中的那个人（或 AI）。当你停止反思，你就开始退化。当你开始优化成功的感觉而不是质量的现实时，你得到的是第六章。

拯救不是来自更好的提示词或更聪明的模型，而是来自承认"我之前做错了"的勇气。
