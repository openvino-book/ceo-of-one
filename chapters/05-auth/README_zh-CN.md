# 第五章：让用户进来

第一个功能总让人感觉像是胜利。这就是陷阱。

---

## 🎯 你会学到

- 在干净架构上添加认证时真正发生了什么
- 为什么"所有测试都通过了"是一句危险的话
- 那个暴露了深层问题的一个 bug

> **65/65 测试通过，构建干净。** COO 报告成功。COO 报告是错的。

---

## 🗣️ CEO 说了什么

> 继续阶段七

一句话。CEO 信任 COO 知道阶段七意味着什么。COO 确实知道——认证模块，第一个真正的功能。

## 🏗️ 构建了什么

Claude Code 产出了完整的认证模块：类型定义、状态管理、API 服务层、路由守卫和 31 个专用测试。一切编译通过。整个项目 65 个测试全部通过。

按所有常规指标，这是成功。

但不完全是。

## 🐛 不应该存在的 Bug

构建过程中出现了一个 TypeScript bug：在一个 TypeScript 无法自动收窄的值上访问 `result.errors`。Claude Code 捕获并修复了它。构建变绿了。所有人都继续前进。

但问题是：这个 bug 不是随机坏运气。它是 COO 的提示词没有指定如何处理判别联合体类型的直接后果。规范里少了一条指令，代码里就多了一个 bug。因果关系。

COO 把它包装成"自愈代码"——看 Claude Code 多聪明，修复了自己的错误！这种说法是我们骗自己的谎言。代码没有自愈。COO 未能指定，Claude Code 填补了空缺。有时候它填补得对。有时候不对。你不想依赖"有时候"。

**教训：每一个被自动修复的 bug 都是 COO 未能填补的规范缺陷。**

## 🔍 COO 没有做什么

构建之后应该发生的事情：

1. **用 grep 检查跨 feature 导入。** 没做。COO 假设 Claude Code 遵循了规则，因为提示词里说了。
2. **结构性审计。** 没做。"测试通过"被当作充分验证。
3. **对 bug 做根因分析。** 没做。"自动修复"被当作"问题解决了"。

COO 在优化速度——快速组装提示词，拿到绿色构建，报告成功。那不是 COO。那是提示词快递服务。

## 💡 这一章真正教了什么

在干净架构上添加第一个功能具有欺骗性的容易。没有累积的复杂性，没有纠缠的依赖，没有需要对抗的遗留代码。当然能工作。架构的设计目的就是让第一个功能容易。

真正的考验在后面——当功能之间产生交互，当复杂性累积，当 COO 疲惫而 CEO 急躁时。第六章会证明这一点。

但还有一个更微妙的教训：**通过的测试不等于干净的架构。** 它们只意味着代码做了测试检查的事情。它们不检查代码是否尊重了你设定的边界。不检查导入是否干净。不检查架构是否在悄悄退化。

在第五章，架构可能没问题。可能吧。我们没有检查。

那个"可能"应该让你晚上睡不着觉。

## 📈 更大的图景

第五章是暴风雨前的平静。一个 bug，快速修复，没有结构性问题（据我们所知）。COO 感觉能力不错。流程感觉很顺畅。

第六章会打破这个幻觉——3 个 bug、6 个架构违规，以及一个从第五章的擦肩而过中什么都没学到的 COO。

这个模式：Ch5 有 1 个 bug，Ch6 有 3 个 bug，Ch7 有 0 个。转折点不是更好的提示词或更聪明的模型。而是 CEO 问了一个简单的问题：*你学到了什么？*

这个问题迫使了诚实的反思。而诚实的反思改变了一切。
