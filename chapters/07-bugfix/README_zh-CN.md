## 🔍 审计

第六章有个让人不舒服的事实：所有测试都通过了。94 个全绿。

但架构被违反了。

支付模块直接导入了课程模块 — 六处，分布在每个文件里。单向依赖规则（feature 只能依赖共享库，不能依赖其他 feature）被打破了。功能测试抓不到这个问题。它们测试的是代码**做了什么**，而不是代码**结构对不对**。

COO 用一个简单的操作发现了这个问题：结构审计。不只跑测试，而是 grep 搜索跨 feature 导入。一条命令，全景呈现。

这是值得养成的习惯。每个功能完成后，在宣布完成之前，问自己：*结构干净吗？* 测试告诉你代码能跑。结构检查告诉你代码是*对的*。

审计还发现了一个附赠 bug：auth store 里的 `enrolledCourses` 从未被初始化。不是立即崩溃的那种，但迟早会出事。

## 🏗️ 修复

发现结构问题后，你有选择。COO 考虑了三种方案：外观模式、事件系统、依赖注入。

外观模式会增加一层新的抽象 — 更多代码、更多复杂度、更多可能出错的地方。事件系统能完全解耦模块，但代价是数据流变得不可见、难以调试。

依赖注入是最无聊的选择。支付模块不再自己去找课程模块拿数据，而是通过函数参数接收它需要的东西。支付模块说："我需要一个查找课程的方法，给我一个能做这件事的函数。" 调用支付代码的人负责提供这个函数。

改动很小。收益巨大。支付模块不再知道课程模块的存在。它们变成了陌生人，通过一个共同朋友（API 路由适配器，一层薄薄的胶水代码，存在于 feature 边界之外）传递纸条。

还有一个没人计划到的副产品：可测试性提升了。因为支付模块现在从外部接收依赖，测试可以传入 mock 数据而非真实 store。测试变得更快、更独立、对实际测试内容的描述也更诚实。

## 💡 零 Bug

看看各章的 bug 趋势：

- 第五章（认证）：1 个 bug
- 第六章（支付）：3 个 bug
- 第七章（修复）：0 个 bug

差别不在于天赋，在于流程。

第七章里，COO 做了一件之前没做过的事：在写规格之前**彻底检查了问题**。等到工程师收到指令时，没有任何歧义。精确的函数签名。精确的文件权限 — 哪些文件能改、哪些不能。精确的验证命令。

当规格精确到这种程度，工程师就没有犯错的空间。零 bug 不值得惊叹 — 它是消除歧义后的自然结果。

换个角度想：如果你告诉别人"修一下支付模块"，你会得到有创意的方案和有创意的 bug。如果你告诉别人"在 `service.ts` 中，让 `createOrder` 接受一个 `findCourse` 参数，类型是 `(id: string) => Promise<Course | null>`，并且不要修改 `courses/` 下的任何文件"，你就会得到你要求的东西。

指令精确是一种善意。它节省所有人的时间。

## 🎓 本章收获

最重要的能力不是构建东西，而是**发现问题并干净地修复它**。

任何人都能添加功能。真正的能力是看着能跑的代码说："这通过了所有测试，但结构不对。" 然后有纪律地在它变成别人的紧急事故之前修好它。

结构审计成本极低。一条 grep 命令五秒钟，能省下后面数小时的调试。把它加入你的 VERIFY 步骤。每次都要。

零 bug 不是要追求的目标 — 它是流程有效的信号。当 COO 先检查再指定，当规格精确到消除歧义，零 bug 就成了默认结果。
