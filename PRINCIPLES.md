## 二、不可违背的核心原则

### 原则 1：每章必须产出真实可运行的东西

- 没有任何一章是纯理论
- 读者看完每一章，手上都有一个能摸得到的结果
- 如果一个章节不能产出实物，砍掉或合并

### 原则 2：设计→验证→总结→写作

每个章节必须严格经过这个循环：

```
设计（假设）→ 验证（新建真实项目做一遍）→ 总结（真知灼见）→ 写作（基于真实经历）
```

**禁止**：凭想象写教程。
**禁止**：复制网上现有资料凑内容。
**必须**：自己先做一遍，拿到结果，然后才写。

### 原则 3：内容必须来自亲身实践

- 每章的洞察 = "我做过了才知道是这样"
- 每章的话术 = "我试过有效才推荐"
- 每章的翻车实录 = "我真实踩过的坑"
- 如果没验证过，标注为"待验证"，先验证再写

### 原则 4：读者不需要学编程

- 读者全程只需要"说话"
- 代码、测试、部署等技术细节是 OpenClaw 和 Claude Code 的事
- 教程正文中不教代码，只教"怎么说"
- 技术细节放在附录，供好奇的读者翻阅

### 原则 5：以终为始

- 终极目标：普通人能做出能赚钱的生产级产品
- 每一章都在向这个目标推进
- 每章开头的"你将收获"必须与这个终极目标有明确关联

### 原则 7：只使用 acpx 方案

- 本项目**只使用 acpx** 指挥 Claude Code，**禁止使用 coding-agent skill 的 `claude --print` 方案**
- acpx 通过 ACP 协议提供结构化 JSON-RPC 输出，让 OpenClaw COO 能**程序化地**精确知道工程师做了什么（tool calls、diffs、测试结果）
- `claude --print` 输出的是人类可读的 ANSI 文本，COO 无法可靠地程序化解读
- 结构化信息是产品品质的保证——COO 能精确验收每个模块
- 工具链：`acpx --approve-all --allowed-tools "..." claude exec "任务"`

### 原则 6：中英双语

- 每个 README 和实验记录都必须有中英双语版本
- 两个版本之间用 badge 形式的链接互相切换
- 文件命名：`README.md`（英文）+ `README_zh-CN.md`（中文）
- 实验记录：`experiment.md`（英文）+ `experiment_zh-CN.md`（中文）

## 三、角色分工

```
CEO（你，读者）
  │  "我要做一个知识付费平台"
  │  只做决策、看效果、说"行"或"不行"
  │
  ▼
COO（OpenClaw）
  │  理解 CEO 意图 → 判断可行性 → 拆解任务
  │  → 分配给 Claude Code → 验收质量 → 汇报进度
  │  同时兼任产品总监——帮 CEO 把关"值不值得做"
  │  不写代码，只指挥
  │
  ▼
开发工程师（Claude Code）
  │  接任务、写代码、跑测试、改 bug
  │  不做产品决策，只做执行
  │
  ▼
CEO（你）
  ← "做好了，你看看效果"
```

## 七、每章的写作结构

```
🎯 你将收获（一句话）
📖 场景故事（真实场景，2 分钟）
🗣️ 指挥话术（核心：你该说什么 + 为什么）
📋 完整对话实录（从你说话到产品完成的真实记录）
🔍 拆解（逐句分析：为什么有效，换个说法会怎样）
⚠️ 翻车实录（真实失败 + 正确改法）
📝 话术模板（可直接复制）
➡️ 下一章预告
```

## 八、Showcase（学员作品展示）

项目核心板块。每个学完第 10 章的读者，可以提交 PR 贡献自己的毕业作品。

**Showcase 的价值：**
- 对新读者：最强的说服力——"跟我一样的普通人做到了"
- 对 GitHub Trending：每个学员 PR = star + fork
- 对项目：永远有新的真实案例，保持活力
